Executive Summary

Replace the in-memory TF-IDF inverted index in lib/text_rank.py with a per-project SQLite WAL-mode database (recall.db) using only Python stdlib sqlite3 + FTS5. Zero new dependencies for Tier 1.

Design Philosophy

1. Zero mandatory dependencies

Any Python 3.8+ install on macOS/Linux/Windows gets BM25 search at session 1 without a pip install. FTS5 is compiled into CPython-bundled SQLite on all major platforms since 2015. Detection: single CREATE VIRTUAL TABLE USING fts5 probe on :memory:.

2. Backward compatibility is absolute

The v3 recall-index.json is byte-for-byte compatible with what the Rust binary and cross-project picker expect. All four Rust-required root keys preserved: sessions, failure_patterns, learnings, pending_learnings. user_messages confirmed absent from v2 stubs and stays absent in v3.

3. recall.db is a derived cache, not the primary store

For sessions 1–keep_count, the sidecar JSON is source of truth; SQLite is the search index. For sessions beyond keep_count (sidecars may be deleted), SQLite is the sole record. synchronous=NORMAL is appropriate because data was fully written before deletion.

SQLite Schema (recall.db)

CREATE TABLE sessions (
  id              TEXT PRIMARY KEY,
  date            TEXT NOT NULL,
  summary         TEXT,
  message_count   INTEGER DEFAULT 0,
  command_count   INTEGER DEFAULT 0,
  failure_count   INTEGER DEFAULT 0,
  topics_text     TEXT DEFAULT '',
  skills_text     TEXT DEFAULT '',
  failures_text   TEXT DEFAULT '',    -- category + error[:80] + resolution[:80]
  learnings_text  TEXT DEFAULT '',
  failures_json   TEXT DEFAULT '[]',  -- scrubbed JSON for display beyond keep_count
  importance      REAL DEFAULT 0.5 CHECK(importance BETWEEN 0.0 AND 1.0),
  evergreen       INTEGER DEFAULT 0 CHECK(evergreen IN (0,1)),
  last_accessed_at TEXT
);

CREATE VIRTUAL TABLE sessions_fts USING fts5(
  summary, topics_text, failures_text, skills_text, learnings_text,
  content='sessions', content_rowid='rowid', tokenize='porter unicode61'
);

-- THREE MANDATORY EXTERNAL-CONTENT FTS5 TRIGGERS (rowid-based)
CREATE TRIGGER sessions_ai AFTER INSERT ON sessions BEGIN
  INSERT INTO sessions_fts(rowid, summary, topics_text, failures_text, skills_text, learnings_text)
  VALUES(new.rowid, new.summary, new.topics_text, new.failures_text, new.skills_text, new.learnings_text);
END;
CREATE TRIGGER sessions_ad AFTER DELETE ON sessions BEGIN
  INSERT INTO sessions_fts(sessions_fts, rowid, summary, topics_text, failures_text, skills_text, learnings_text)
  VALUES('delete', old.rowid, old.summary, old.topics_text, old.failures_text, old.skills_text, old.learnings_text);
END;
CREATE TRIGGER sessions_au AFTER UPDATE ON sessions BEGIN
  INSERT INTO sessions_fts(sessions_fts, rowid, summary, topics_text, failures_text, skills_text, learnings_text)
  VALUES('delete', old.rowid, old.summary, old.topics_text, old.failures_text, old.skills_text, old.learnings_text);
  INSERT INTO sessions_fts(rowid, summary, topics_text, failures_text, skills_text, learnings_text)
  VALUES(new.rowid, new.summary, new.topics_text, new.failures_text, new.skills_text, new.learnings_text);
END;

CREATE TABLE failure_patterns (category TEXT, cmd TEXT, error TEXT, resolution TEXT, session_id TEXT, date TEXT);
CREATE TABLE learnings        (bucket TEXT, category TEXT, title TEXT, fix TEXT, solution TEXT, session_id TEXT, date TEXT);
CREATE TABLE entities         (entity TEXT, entity_type TEXT, session_id TEXT, PRIMARY KEY(entity, entity_type, session_id));
CREATE TABLE usage_stats      (session_id TEXT PRIMARY KEY, last_accessed TEXT, access_count INTEGER DEFAULT 0);
CREATE TABLE db_meta          (key TEXT PRIMARY KEY, value TEXT);

CREATE INDEX idx_sessions_date       ON sessions(date DESC);
CREATE INDEX idx_sessions_importance ON sessions(importance DESC);
CREATE INDEX idx_entities_entity     ON entities(entity);

Search Algorithm (/recall <term>)

Total latency at 1000 sessions: ~4ms. At 10,000 sessions: ~12ms.

# Step 1: FTS5 probe (once per process, cached in _fts5_available)
probe = sqlite3.connect(':memory:')
probe.execute('CREATE VIRTUAL TABLE _probe USING fts5(x)')
# Falls back to lib/text_rank.py TF-IDF if this raises OperationalError

# Step 2: Open with WAL + optional fcntl fallback (db_open_write helper)
actual_mode = db.execute('PRAGMA journal_mode=WAL').fetchone()[0]
if actual_mode != 'wal':
    fcntl.flock(lockfd, fcntl.LOCK_EX)  # NFS/Docker: process-safe advisory lock

# Step 3: Health check FTS5 index (once per process)
db.execute("SELECT count(*) FROM sessions_fts "
           "JOIN sessions s ON s.rowid = sessions_fts.rowid "  # JOIN required!
           "WHERE sessions_fts MATCH 'test' LIMIT 0")
# On failure: INSERT INTO sessions_fts(sessions_fts) VALUES('rebuild')

# Step 4: BM25 primary search + importance UNION (always; <2ms at 1k sessions)
rows_bm25 = db.execute('''
  SELECT s.id, s.date, s.summary, s.importance, s.evergreen, s.failures_json,
         bm25(sessions_fts, 3.0, 2.0, 1.0, 1.5, 1.5) AS bm25_score
  FROM sessions_fts
  JOIN sessions s ON s.rowid = sessions_fts.rowid   # ALWAYS JOIN — never bare MATCH
  WHERE sessions_fts MATCH ?
  ORDER BY bm25_score LIMIT 80
''', (query,))
# + UNION with top-20 by importance (eliminates hard retrieval ceiling)

# Step 5: Entity co-occurrence boost (<1ms)
entity_ids = {r[0] for r in db.execute(
    'SELECT DISTINCT session_id FROM entities WHERE entity IN (?,...)', query_tokens)}

# Step 6: Python scoring — 180-day half-life + importance floor (~0.4ms for 100 rows)
decay = 1.0 if row['evergreen'] else math.exp(-0.00385 * age_days)
# k=0.00385 → half-life=180 days; floor = importance * 0.25 (activates ~435 days)
effective_decay = max(decay, imp * 0.25)
entity_boost    = 1.15 if row['id'] in entity_ids else 1.0
combined        = bm25_norm * effective_decay * (0.8 + 0.2 * imp) * entity_boost

# Step 7: Update usage_stats (write path only; NEVER at SessionStart where query_only=1)
db.execute('INSERT INTO usage_stats ... ON CONFLICT DO UPDATE SET access_count=access_count+1')

# Step 8: Load top-8 session sidecars; fallback to failures_json blob for sessions 101+

# Step 9: WAL checkpoint (SessionEnd only)
db.execute('PRAGMA wal_checkpoint(FULL)')  # falls back to PASSIVE on concurrent reader

Write-pattern compatibility (important!)

Write patternSafe?Note
INSERTSAFEAI trigger fires
UPDATESAFEAU trigger fires (delete + insert)
DELETESAFEAD trigger fires tombstone
INSERT OR REPLACESAFE WITH JOIN ONLYBare MATCH (no JOIN) returns stale orphan counts. Always use JOIN guard. Avoid in write paths — rowid churn inflates WAL.
INSERT...ON CONFLICT DO UPDATESAFEAU trigger fires on UPDATE branch

Hook Implementations

SessionStart (~17ms total)

# Phase 1: Rust fast-path (unchanged; ~5ms)
# Reads recall-index.json v3; format identical to v2; exits 64 if no notable context

# Phase 2: Python enrichment (~12ms; runs on Rust exit 64 or RECALL_SESSION_START_VERBOSE=1)
db = sqlite3.connect(str(recall_db_path))
db.execute('PRAGMA query_only=1')  # enforced read-only; usage_stats NEVER written here

# Recent sessions (date index, no FTS; <1ms)
SELECT id, date, summary, topics_text, failure_count, importance
FROM sessions ORDER BY date DESC LIMIT 5

# Recurring failures (last 30 days; <1ms)
SELECT category, COUNT(*) as cnt, MAX(resolution) as resolution
FROM failure_patterns WHERE date > date('now', '-30 days')
GROUP BY category HAVING cnt >= 2 ORDER BY cnt DESC LIMIT 5

# Output: "Recall: 47 sessions; last 2026-03-20. Topics: payroll, rounding."
# If recall.db absent: falls back to v3 JSON stubs (same code path as pre-migration)

SessionEnd — v3 write path detection

index = load_index(project_folder, create_if_missing=True)
recall_db = get_project_dir(project_folder) / 'recall.db'

if index.get('version', 0) >= 3 and recall_db.exists():
    db = db_open_write(recall_db)  # WAL + optional fcntl lock
    # explicit INSERT or UPDATE (never INSERT OR REPLACE)
    db.execute('INSERT INTO sessions (id, ...) VALUES (?, ...)', _build_session_row(session_data))
    db.commit()
    db.execute('PRAGMA wal_checkpoint(FULL)')
    _refresh_v3_pointer_file(index, project_folder, session_id, session_data)
else:
    # v2 JSON path — unchanged for unmigrated projects
    index['sessions'][session_id] = create_session_summary(session_data)
    save_index(project_folder, index)

Tiered Architecture

Tier 1 Ships Now

Zero new dependencies · Python stdlib only

  • FTS5 BM25 multi-column keyword search
  • Porter stemming + unicode61 tokenizer
  • failures_text with actual error text (not just categories)
  • 180-day recency decay + importance floor
  • UNION retrieval: BM25 top-80 + importance top-20
  • Entity co-occurrence boost (1.15×)
  • /recall doctor diagnostic command
  • ~4ms at 1k sessions · ~3.5KB/session storage

Tier 2 Separate PR · Opt-in

Gated: RECALL_EXPERIMENTAL_EMBEDDINGS=1

  • sqlite-vec semantic vector search
  • fastembed ONNX (~80MB model, no PyTorch)
  • Hybrid BM25 + cosine similarity re-ranking
  • /recall embed-all backfill command
  • macOS: requires pysqlite3-binary workaround
  • ~60ms total search latency
  • Tier 1 fully functional when Tier 2 enabled

Migration Plan (v2 → v3)

Never automatic. Run: python3 migrations/v2_to_v3_fts5.py (dry-run by default) or --apply to execute. Matches migrations/v1_to_v2_restart.py convention. Under 3 seconds at 1000 sessions.

Migration steps

# Step 3: CREATE recall.db with full schema (all DDL above)

#         Per-session error handling: skip malformed sessions, log + continue

# Step 7: Write v3 recall-index.json atomically (last-50 stubs, v2-identical format)

# Step 9: Verify counts match; FTS5 smoke test; Rust binary compatibility test
# Step 10: Remove sentinel
# Rollback on any exception: db.rollback() + db_path.unlink() + sentinel.unlink()

Note: Karpathy LLM Wiki Pattern (Tier 3 Opportunity)

What the pattern proposes

Karpathy's gist describes maintaining a persistent wiki of LLM-generated markdown pages alongside raw sources. Instead of querying raw session records, agents query distilled, cross-referenced knowledge pages. Three layers: raw sources (immutable) → wiki (LLM-maintained markdown) → schema (config). Three operations: Ingest (new source → update 5–10 wiki pages), Query (hit wiki not raw sources), Lint (health-check for contradictions/orphans).

Karpathy layerrecall equivalent
Raw sources (immutable).jsonl + recall-sessions/{uuid}.json
The wiki (LLM-maintained markdown)New: recall-wiki/ — one page per file/topic/pattern
Schema (config)Skill config / learnings system

How it maps to recall hooks

Ingest = SessionEnd hook: after FTS5 index write, LLM updates 5–10 relevant wiki pages (recall-wiki/src_payroll.md, recall-wiki/pytest_errors.md).
Query = /recall <term>: FTS5 finds the relevant wiki pages; wiki content injected instead of raw session stubs.
Lint = /recall cleanup: detect contradictions, stale pages, orphaned cross-references.

The key difference: FTS5 retrieves which sessions touched a topic. The wiki retrieves what is known about the topic — cross-session, distilled, LLM-friendly markdown. "Here's everything recall knows about JWT" vs "here are 3 sessions mentioning JWT".

This is a natural Tier 3 on top of the FTS5 spec. The FTS5 index becomes the navigation layer; the wiki becomes the knowledge layer.
