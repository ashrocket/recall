# recall v3 — Index Format Specification

**recall v3 Index Format Specification: FTS5-First SQLite Hybrid (Production-Hardened)**

*Produced by: 6-area research sprint → 3 initial designs → 3 rounds of (5-reviewer panel + 3-writer revision) → synthesis*
*Date: 2026-06-21*

---

## Executive Summary

Replace the in-memory TF-IDF inverted index in lib/text_rank.py with a per-project SQLite WAL-mode database (recall.db) using only Python stdlib sqlite3 and FTS5. The design is tiered: Tier 1 (this release) requires zero new dependencies and delivers BM25 multi-column keyword search with 180-day recency decay and importance scoring in under 5ms at 1000 sessions. Tier 2 (follow-on PR) adds optional semantic vector search via sqlite-vec and fastembed. The existing recall-index.json becomes a v3 pointer file retaining the last 50 session stubs in v2-identical format, preserving full Rust fast-path compatibility with zero Rust code changes. Migration is always opt-in via a standalone script. Five standalone patches ship first and benefit all v2 users immediately: atomic save_index(), failures_text enrichment, scrub_for_index() implementation, RECALL_VERSION constant, and the /recall doctor fast-binary bypass fix.

---

## Design Philosophy

Three principles govern every decision. First, zero mandatory dependencies: any Python 3.8+ install on macOS/Linux/Windows gets BM25 search at session 1 without a pip install. FTS5 is compiled into CPython-bundled SQLite on all major platforms since 2015; detection is a single :memory: CREATE VIRTUAL TABLE probe with zero side effects. Second, backward compatibility is absolute: the v3 recall-index.json is byte-for-byte compatible with what the Rust binary and cross-project picker expect; user_messages is confirmed absent from v2 stubs (empirically verified from create_session_summary() in session-end.py lines 282-291 and codex_session_end.py lines 54-64) and stays absent in v3; all four Rust-required root keys (sessions, failure_patterns, learnings, pending_learnings) are preserved. Third, recall.db is a derived cache, not the primary store: for sessions 1 through approximately keep_count, the sidecar JSON is the source of truth and SQLite is the search index; for sessions beyond keep_count (where sidecars may have been probabilistically deleted), SQLite is the sole record and synchronous=NORMAL is appropriate because the index was derived from source data that was fully written before deletion.

## Index Files on Disk

### `~/.claude/projects/{project_folder}/recall.db`
**Purpose:** SQLite WAL-mode database. Tier 1 provides FTS5 BM25 keyword search with Porter stemming. Tier 2 (deferred) adds vector embeddings via sessions_vec table. Binary format; inspect via sqlite3 CLI. Sole record for sessions beyond approximately RECALL_SIDECAR_KEEP_COUNT (default 100) due to probabilistic 10% cleanup trigger — actual deletion threshold is non-deterministic between session 100 and ~1000.

**Format:** SQLite WAL (journal_mode=WAL, synchronous=NORMAL by default; RECALL_SYNCHRONOUS_FULL=1 env var enables synchronous=FULL for users needing OS-crash durability for sessions beyond keep_count)

### `~/.claude/projects/{project_folder}/recall-index.json`
**Purpose:** v3 pointer file. Retains last 50 session stubs in full v2-identical format for Rust fast-path and cross-project picker. Also stores failure_patterns, learnings, pending_learnings, usage for Rust and Python consumers that read these root keys. Version 3 adds top-level keys: 'db': 'recall.db' and 'minimum_reader_version': '3.0.0'. Written atomically via os.replace() from .json.tmp.

**Format:** JSON, indented, written atomically. MAX_SESSIONS_IN_INDEX=50 stubs, MAX_INDEX_SIZE_KB=60 limit enforced by prune_index()

### `~/.claude/projects/{project_folder}/recall-sessions/{uuid}.json`
**Purpose:** Full session sidecar. Source of truth for sessions 1 through approximately keep_count. Contains all fields including user_messages (up to 30), commands (up to 50), failures (up to 20), skills_used. Retained for the most recent RECALL_SIDECAR_KEEP_COUNT sessions (default 100, recommended 200); cleanup_old_detail_files() is triggered probabilistically (~10% chance per SessionEnd). Human-readable plaintext; users may inspect with any editor.

**Format:** JSON, indented, human-readable

### `~/.claude/projects/{project_folder}/.recall-migrating`
**Purpose:** Sentinel file written at migration start and deleted on completion or rollback. Presence causes codex_session_end.py to skip index write and exit 0 cleanly. Absence is the normal operating state.

**Format:** Plain text containing ISO timestamp of migration start

### `~/.claude/projects/{project_folder}/.recall-write.lock`
**Purpose:** fcntl advisory lock target file for NFS/Docker environments where WAL mode cannot engage (PRAGMA journal_mode=WAL returns a mode other than 'wal'). All writers must acquire LOCK_EX via db_open_write() helper in lib/shared.py before any SQLite write. Advisory only — all writers must cooperate. Absent on local filesystems where WAL works normally.

**Format:** Empty or minimal text file; only existence matters for flock target

### `~/.claude/projects/{project_folder}/recall-index.json.v2.bak`
**Purpose:** Backup of the v2 recall-index.json written atomically before migration completes. Rollback target if migration fails. Never overwritten if present (migration is idempotent).

**Format:** JSON, identical to pre-migration recall-index.json

## Search Implementation (`/recall <term>`)

"Step-by-step search algorithm for: /recall payroll rounding\n\nMODULE-LEVEL STATE (initialized once at import time):\n  _fts5_available: Optional[bool] = None   # None = not yet checked\n  _fts5_health_checked: bool = False        # process-level gate; health check runs once per process\n\nSTEP 1: FTS5 AVAILABILITY DETECTION (once per process, cached)\n  if _fts5_available is None:\n      try:\n          probe = sqlite3.connect(':memory:')  # :memory: ONLY — never probe on real db file\n          probe.execute('CREATE VIRTUAL TABLE _probe USING fts5(x)')\n          probe.close()\n          _fts5_available = True\n      except sqlite3.OperationalError:\n          _fts5_available = False  # fall back to lib/text_rank.py TF-IDF unchanged\n\n  if not _fts5_available or not recall_db_path.exists():\n      return _tfidf_fallback(query, project_folder)  # zero regression\n\nSTEP 2: OPEN DATABASE VIA db_open_write() HELPER (~0.5ms)\n  # db_open_write() is a centralized helper in lib/shared.py.\n  # All writers MUST use this helper — the advisory lock is enforced by convention, not OS.\n  def db_open_write(db_path: Path):\n      db = sqlite3.connect(str(db_path))\n      db.row_factory = sqlite3.Row\n      actual_mode = db.execute('PRAGMA journal_mode=WAL').fetchone()[0]\n      if actual_mode != 'wal':\n          sys.stderr.write('recall: WAL unavailable (NFS/Docker?); using fcntl advisory lock\\n')\n          import fcntl\n          lockfd = open(db_path.parent / '.recall-write.lock', 'w')\n          fcntl.flock(lockfd, fcntl.LOCK_EX)\n          # caller is responsible for releasing lock after db.commit()\n          db._lockfd = lockfd  # stash for caller to release\n      synchronous = 'FULL' if os.environ.get('RECALL_SYNCHRONOUS_FULL') == '1' else 'NORMAL'\n      db.execute(f'PRAGMA synchronous={synchronous}')\n      db.execute('PRAGMA cache_size=-16000')\n      return db\n\nSTEP 3: BOOTSTRAP-SAFE HEALTH CHECK (once per process)\n  if not _fts5_health_checked:\n      _fts5_health_checked = True\n      tbl = db.execute(\n          \"SELECT name FROM sqlite_master WHERE type='table' AND name='sessions_fts'\"\n      ).fetchone()\n      if tbl:\n          try:\n              # JOIN form REQUIRED — bare MATCH without JOIN returns stale counts on\n              # orphan rowids left by INSERT OR REPLACE (empirically verified: bare MATCH\n              # returns 2 after one INSERT OR REPLACE on SQLite 3.51.0, JOIN returns 1)\n              db.execute(\n                  \"SELECT count(*) FROM sessions_fts \"\n                  \"JOIN sessions s ON s.rowid = sessions_fts.rowid \"\n                  \"WHERE sessions_fts MATCH 'test' LIMIT 0\"\n              )\n          except sqlite3.OperationalError:\n              db.execute(\"INSERT INTO sessions_fts(sessions_fts) VALUES('rebuild')\")\n              n = db.execute('SELECT COUNT(*) FROM sessions').fetchone()[0]\n              sys.stderr.write(\n                  f'recall: FTS5 index rebuilt from sessions table ({n} sessions). '\n                  'No data loss — sessions table is intact. '\n                  'Sidecar files may be absent for sessions beyond keep_count '\n                  'but all sessions remain indexed in SQLite.\\n'\n              )\n\nSTEP 4: FTS5 BM25 PRIMARY SEARCH + IMPORTANCE UNION (always; total <2ms at 1000 sessions)\n  # Column weights: summary=3x, topics_text=2x, failures_text=1x, skills=1.5x, learnings=1.5x\n  # bm25() returns negative floats; ORDER BY ASC (most negative = most relevant)\n  # LIMIT 80 for BM25 results; UNION with top-20 by importance ensures high-value sessions\n  # are always candidates regardless of BM25 rank (eliminates hard retrieval ceiling)\n  rows_bm25 = db.execute('''\n    SELECT s.id, s.date, s.summary, s.importance, s.evergreen, s.last_accessed_at,\n           s.failures_json,\n           bm25(sessions_fts, 3.0, 2.0, 1.0, 1.5, 1.5) AS bm25_score\n    FROM sessions_fts\n    JOIN sessions s ON s.rowid = sessions_fts.rowid\n    WHERE sessions_fts MATCH ?\n    ORDER BY bm25_score\n    LIMIT 80\n  ''', (query,)).fetchall()\n\n  bm25_ids = {r['id'] for r in rows_bm25}\n  # Importance boost: add top-20 by importance that BM25 missed\n  placeholder = ','.join('?' * len(bm25_ids)) if bm25_ids else '?'\n  params = list(bm25_ids) if bm25_ids else ['__none__']\n  rows_importance = db.execute(f'''\n    SELECT s.id, s.date, s.summary, s.importance, s.evergreen, s.last_accessed_at,\n           s.failures_json, NULL AS bm25_score\n    FROM sessions s\n    WHERE s.importance >= 0.65 AND s.id NOT IN ({placeholder})\n    ORDER BY s.importance DESC, s.date DESC\n    LIMIT 20\n  ''', params).fetchall()\n\n  rows_all = list(rows_bm25) + list(rows_importance)\n\nSTEP 5: ENTITY CO-OCCURRENCE BOOST (<1ms)\n  # Filter tokens: length >= 3 avoids stop words producing spurious joins\n  query_tokens = [t.lower() for t in query.split() if len(t) >= 3]\n  entity_ids: set = set()\n  if query_tokens:\n      placeholders = ','.join('?' * len(query_tokens))\n      entity_ids = {r[0] for r in db.execute(\n          f'SELECT DISTINCT session_id FROM entities WHERE entity IN ({placeholders})',\n          query_tokens\n      ).fetchall()}\n\nSTEP 6: PYTHON-SIDE SCORING WITH 180-DAY HALF-LIFE AND IMPORTANCE FLOOR (~0.4ms for 100 rows)\n  # k = 0.00385 gives half-life of 180 days: exp(-0.00385 * 180) ≈ 0.5\n  # Floor activates at approximately 435 days (1.2 years) for importance=0.75:\n  #   exp(-0.00385 * d) < 0.75 * 0.25 = 0.1875  =>  d > 435 days\n  # Floor prevents complete burial for high-importance sessions in year 2+.\n  # A moderately-relevant recent session WILL correctly outscore a highly-relevant\n  # 1-year-old session in realistic corpus conditions — this is correct behavior.\n  import math, datetime\n  now = datetime.datetime.now()\n  results = []\n  for row in rows_all:\n      age_days = (now - datetime.datetime.fromisoformat(row['date'])).days\n      decay = 1.0 if row['evergreen'] else math.exp(-0.00385 * age_days)\n      imp = row['importance'] if row['importance'] is not None else 0.5\n      effective_decay = max(decay, imp * 0.25)   # importance floor\n      if row['bm25_score'] is not None:\n          bm25_norm = 1.0 / (1.0 + abs(row['bm25_score']))\n      else:\n          bm25_norm = 0.15  # importance-only candidates get a low base BM25 signal\n      entity_boost = 1.15 if row['id'] in entity_ids else 1.0\n      combined = bm25_norm * effective_decay * (0.8 + 0.2 * imp) * entity_boost\n      results.append((combined, row))\n  results.sort(key=lambda x: x[0], reverse=True)\n\nSTEP 7: UPDATE USAGE STATS (write-only; NEVER at SessionStart where PRAGMA query_only=1)\n  # Write to usage_stats only; sessions.last_accessed_at updated lazily in periodic sync.\n  # This eliminates AU trigger cascade during interactive /recall reads.\n  top_ids = [r[1]['id'] for r in results[:8]]\n  now_iso = now.isoformat()\n  for sid in top_ids:\n      db.execute('''\n        INSERT INTO usage_stats(session_id, last_accessed, access_count) VALUES(?,?,1)\n        ON CONFLICT(session_id) DO UPDATE SET\n          last_accessed=excluded.last_accessed,\n          access_count=access_count+1\n      ''', (sid, now_iso))\n  db.commit()\n  if hasattr(db, '_lockfd'):\n      import fcntl\n      fcntl.flock(db._lockfd, fcntl.LOCK_UN)\n      db._lockfd.close()\n\nSTEP 8: LOAD FULL DETAILS AND FORMAT (top-8 sessions)\n  import json as _json\n  for score, row in results[:8]:\n      sidecar = load_session_details(project_folder, row['id'])\n      if sidecar is None:\n          # Graceful fallback: use SQLite row (failures_json provides structured display\n          # for sessions beyond keep_count where sidecar was deleted)\n          sidecar = dict(row)\n          if row['failures_json']:\n              sidecar['failures'] = _json.loads(row['failures_json'])\n      print_session_result(sidecar, score)\n\nSTEP 9: WAL CHECKPOINT (SessionEnd only, after all writes complete)\n  try:\n      db.execute('PRAGMA wal_checkpoint(FULL)')\n  except sqlite3.OperationalError:\n      db.execute('PRAGMA wal_checkpoint(PASSIVE)')\n      sys.stderr.write('recall: wal_checkpoint(FULL) deferred to PASSIVE (concurrent reader)\\n')\n\nTotal latency at 1000 sessions: FTS5+UNION <2ms + entity join <1ms + Python scoring 0.4ms + sidecar loads ~1ms = ~4ms\n\nWRITE PATTERN COMPATIBILITY TABLE (external-content FTS5, empirically verified SQLite 3.51.0):\n  INSERT:                      SAFE — AI trigger fires, adds FTS5 entry\n  UPDATE:                      SAFE — AU trigger fires delete+insert, rowid preserved\n  DELETE:                      SAFE — AD trigger fires tombstone\n  INSERT OR REPLACE:           SAFE WITH JOIN GUARD ONLY — AD fires at old rowid,\n                               AI fires at new rowid; bare MATCH (no JOIN) returns\n                               stale orphan counts (empirically: returns 2 not 1);\n                               JOIN sessions s ON s.rowid = sessions_fts.rowid\n                               correctly filters orphans. Avoid in session-end.py:\n                               rowid churn inflates WAL. Use INSERT-then-UPDATE.\n  INSERT...ON CONFLICT UPDATE: SAFE — AU trigger fires on UPDATE branch\n  REQUIRED pattern (session-end.py, codex_session_end.py): explicit check-then-INSERT\n  or check-then-UPDATE. Never INSERT OR REPLACE in write paths.\n  NEVER run: SELECT ... FROM sessions_fts WHERE sessions_fts MATCH ?\n  ALWAYS use: SELECT ... FROM sessions_fts JOIN sessions s ON s.rowid=sessions_fts.rowid\n              WHERE sessions_fts MATCH ?"

Sessions dict is format-identical to v2.\n  user_messages absent from stubs in both v2 and v3 — Rust line 162 uses\n  last_session.get('user_messages').and_then(Value::as_array) which safely\n  returns None when absent. ## Tiered Architecture

### Tier 1 — Baseline (zero new dependencies, ships now)

"Tier 1 (this release): stdlib only, zero pip installs.\n\nDetection: CREATE VIRTUAL TABLE _probe USING fts5(x) in sqlite3.connect(':memory:').\nFails gracefully on Anaconda Windows and some conda-forge builds where FTS5 is not\ncompiled in — falls back to lib/text_rank.py TF-IDF unchanged.\n\nCapabilities delivered at Tier 1:\n- FTS5 BM25 multi-column keyword search with Porter stemming and unicode61 tokenizer\n- Per-column relevance weights (summary=3x, topics=2x, failures=1x, skills=1.5x, learnings=1.5x)\n- failures_text indexes actual error message text and resolution strings (not just category tokens)\n  enabling queries like 'AssertionError', 'authentication failed', 'Decimal arithmetic'\n- 180-day recency decay (k=0.00385) with importance floor activating at ~435 days for imp=0.75\n  (empirically computed: floor fires when exp(-0.00385*d) < imp*0.25)\n- UNION retrieval: top-80 BM25 + top-20 by importance, eliminating hard retrieval ceiling\n- Entity co-occurrence boost (1.15x) from topics, skills, error categories stored in entities table\n- /recall doctor diagnostic command (wired through Python dispatch, not Rust fast-path)\n- Structured failure display for sessions beyond keep_count via failures_json blob column\n- Latency: ~4ms at 1000 sessions, ~12ms at 10000 sessions\n- Storage: ~3.5KB per session (sessions row ~550 bytes, FTS5 index ~1.2KB, entities ~280 bytes,\n  sidecar JSON ~1.5KB)\n\nThe Tier 1 FTS5 search is a substantial improvement over boolean-AND substring matching.\nAt fewer than 20 sessions, BM25 term frequency signals are weak and TF-IDF fallback\nmay outperform — this is acceptable and the fallback remains available."

### Tier 2 — Enhanced (separate PR, opt-in)

"Tier 2 (separate follow-on PR): optional semantic vector search.\n\nGating: RECALL_EXPERIMENTAL_EMBEDDINGS=1 environment variable. Never enabled by default.\nNot included in the initial v3 release.\n\nPackages required:\n- sqlite-vec>=0.1.9 (pip install sqlite-vec)\n- fastembed>=0.3.0 (pip install fastembed; ONNX-based, no PyTorch; ~80MB model download)\n- pysqlite3-binary>=0.5.2 (macOS only workaround for enable_load_extension AttributeError\n  confirmed empirically on macOS system Python; Tier 1 FTS5 is entirely unaffected)\n\nArchitecture: adds sessions_vec virtual table (vec0) alongside sessions_fts.\nEmbeddings stored as FLOAT32 blobs. Search becomes hybrid: BM25 score + cosine\nsimilarity from sessions_vec, re-ranked in Python.\n\nPrerequisites before Tier 2 ships:\n- Pinned model identifier with SHA256 hash verification\n- /recall embed-all command for embedding backfill on pre-v3 sessions\n- Documented pysqlite3-binary installation on macOS\n- RECALL_EXPERIMENTAL_EMBEDDINGS=1 gate behavior fully specified\n\nTier 1 remains fully functional and unaffected when Tier 2 is enabled.\nAll FTS5 infrastructure is compatible with Tier 2 addition."

**Required dependencies:** sqlite3 (Python stdlib — FTS5 built into CPython-bundled SQLite on macOS/Linux since 2015; detected via CREATE VIRTUAL TABLE in :memory:; zero install), json (Python stdlib), pathlib (Python stdlib), datetime (Python stdlib), math (Python stdlib — for 180-day recency decay exp() in Python post-query), os (Python stdlib — for os.replace() atomic write in save_index), re (Python stdlib — for scrub_for_index() pattern matching), fcntl (Python stdlib POSIX only — advisory lock when WAL unavailable on NFS/Docker; absent on Windows where local NTFS WAL works normally; document Windows NFS as unsupported for concurrent writes)

**Optional dependencies:** sqlite-vec>=0.1.9 (Tier 2 DEFERRED — pip install sqlite-vec; gated behind RECALL_EXPERIMENTAL_EMBEDDINGS=1; blocked on macOS system Python by enable_load_extension AttributeError confirmed empirically), fastembed>=0.3.0 (Tier 2 DEFERRED — ONNX-based, no PyTorch, ~80MB model download; requires pinned model ID with SHA256 verification before production use), pysqlite3-binary>=0.5.2 (Tier 2 macOS workaround DEFERRED — only needed on macOS system Python for extension loading; Tier 1 FTS5 entirely unaffected)

## Migration Plan (v2 → v3)

"Migration script: migrations/v2_to_v3_fts5.py\nConvention: matches migrations/v1_to_v2_restart.py (dry-run default, --apply to execute)\n\nUsage:\n  python3 migrations/v2_to_v3_fts5.py                     # dry-run (default)\n  python3 migrations/v2_to_v3_fts5.py --apply             # migrate current project\n  python3 migrations/v2_to_v3_fts5.py --apply --project /path  # specific project\n  python3 migrations/v2_to_v3_fts5.py --apply --all       # all projects\n\nNEVER triggered automatically from SessionEnd or SessionStart.\n\nDRY-RUN OUTPUT (per-session, matching v1_to_v2_restart.py style):\n  [DRY RUN] 1/47: 2026-03-20 'Implementing payroll bonus calculation fix' failures=2 importance=0.70\n  [DRY RUN] 2/47: 2026-03-18 'Migrated CI pipeline to GitHub Actions' failures=0 importance=0.50\n  ...\n  [DRY RUN] Would create recall.db with 47 sessions. Run with --apply to execute.\n\nPRE-REQUISITE STANDALONE PATCHES (ship in same commit wave, before migration script):\n  These patches benefit all v2 users immediately and are independent of migration:\n  1. lib/shared.py save_index(): write to index_file.with_suffix('.json.tmp') then os.replace()\n  2. hooks/scripts/session-end.py parse_session_full(): enrich failures_text with\n     error[:80] + resolution[:80] per failure\n  3. lib/sync_scan.py: add scrub_for_index() function and 4 new SECRET_PATTERNS\n  4. lib/shared.py: add RECALL_VERSION = '3.2.1' constant (sync from Cargo.toml)\n  5. bin/recall-sessions.py line 31: add 'doctor' to _early_fast_binary_supports() exclusion set\n  6. hooks/scripts/codex_session_end.py: add sentinel check before save_index() at line ~294\n  7. lib/sync.py gather_sync_files(): exclude *.db, *.db-wal, *.db-shm, .recall-migrating,\n     .recall-write.lock at engine layer as defense-in-depth\n\nMIGRATION STEPS:\n\nSTEP 0: WRITE SENTINEL\n  sentinel = get_project_dir(project_folder) / '.recall-migrating'\n  sentinel.write_text(datetime.now().isoformat())\n\nSTEP 1: DETECT AND LOAD V2 INDEX\n  index = load_index(project_folder)\n  if index is None or index.get('version', 0) >= 3:\n      sentinel.unlink()  # idempotent — already migrated\n      return\n  v2_backup_str = json.dumps(index, indent=2, default=str)\n\nSTEP 2: FTS5 PROBE (fail fast before creating any files)\n  probe = sqlite3.connect(':memory:')\n  try:\n      probe.execute('CREATE VIRTUAL TABLE _p USING fts5(x)')\n  except sqlite3.OperationalError:\n      sentinel.unlink()\n      sys.exit('FTS5 unavailable on this Python install. '\n               'Install Python from python.org or Homebrew.')\n  probe.close()\n\nSTEP 3: CREATE recall.db WITH FULL SCHEMA\n  db_path = get_project_dir(project_folder) / 'recall.db'\n  db = sqlite3.connect(str(db_path))\n  actual_mode = db.execute('PRAGMA journal_mode=WAL').fetchone()[0]\n  db.execute('PRAGMA synchronous=NORMAL')\n  db.execute('PRAGMA cache_size=-16000')\n  # Run all DDL: CREATE TABLE sessions, CREATE VIRTUAL TABLE sessions_fts,\n  # CREATE TRIGGER sessions_ai/ad/au, CREATE TABLE failure_patterns, learnings,\n  # entities, usage_stats, db_meta; INSERT INTO db_meta VALUES;\n  # CREATE INDEX idx_sessions_date, idx_entities_entity\n\nSTEP 4: MIGRATE SESSIONS WITH PER-SESSION ERROR HANDLING\n  from lib.sync_scan import scrub_for_index\n  db.execute('BEGIN')\n  migrated = 0; skipped = 0\n  for session_id, s in index.get('sessions', {}).items():\n      try:\n          topics   = s.get('topics', [])\n          skills   = [x if isinstance(x, str) else x.get('skill','') for x in s.get('skills_used', [])]\n          failures = [f for f in s.get('failures', []) if isinstance(f, dict)]\n          learnings = s.get('learnings', [])\n          if isinstance(learnings, str): learnings = [learnings]\n\n          topics_text    = scrub_for_index(' '.join(t for t in topics if isinstance(t, str)))\n          skills_text    = scrub_for_index(' '.join(sk for sk in skills if sk))\n          # failures_text includes actual error text + resolution (core retrieval improvement):\n          # Both hooks use 'command' for the cmd field and 'error' for error text.\n          failures_text  = scrub_for_index(' '.join(\n              (f.get('category','') + ' '\n               + f.get('error','')[:80] + ' '\n               + f.get('resolution','')[:80]).strip()\n              for f in failures\n          ))\n          learnings_text = scrub_for_index(' '.join(\n              l[:60] if isinstance(l, str) else l.get('title','')[:60] for l in learnings\n          ))\n          summary_clean  = scrub_for_index(s.get('summary', ''))\n          # Scrub failures_json blob (structured display for sessions beyond keep_count)\n          failures_scrubbed = json.dumps([\n              {**f,\n               'command': scrub_for_index(f.get('command', '')),\n               'error':   scrub_for_index(f.get('error', ''))}\n              for f in failures\n          ])\n\n          # Importance calibration (failures-with-resolution > learnings: more searchable)\n          has_resolution = any(f.get('resolution') for f in failures)\n          has_learnings  = bool(learnings)\n          if has_resolution and has_learnings: importance = 0.75\n          elif has_resolution:                  importance = 0.70\n          elif has_learnings:                   importance = 0.65\n          else:                                 importance = 0.50\n\n          # Explicit INSERT only (never INSERT OR REPLACE; avoid rowid churn in WAL)\n          exists = db.execute('SELECT 1 FROM sessions WHERE id=?', (session_id,)).fetchone()\n          if not exists:\n              db.execute(\n                  'INSERT INTO sessions (id, date, summary, message_count, command_count,'\n                  ' failure_count, topics_text, skills_text, failures_text,'\n                  ' learnings_text, failures_json, importance) VALUES (?,?,?,?,?,?,?,?,?,?,?,?)',\n                  (session_id, s.get('date'), summary_clean,\n                   s.get('message_count',0), s.get('command_count',0),\n                   s.get('failure_count',0),\n                   topics_text, skills_text, failures_text, learnings_text,\n                   failures_scrubbed, importance)\n              )  # AI trigger fires automatically\n\n          # Entity extraction (topics, error categories, skills)\n          for topic in topics[:10]:\n              db.execute('INSERT OR IGNORE INTO entities VALUES(?,?,?)',\n                         (topic.lower(), 'topic', session_id))\n          for f in failures:\n              if f.get('category'):\n                  db.execute('INSERT OR IGNORE INTO entities VALUES(?,?,?)',\n                             (f['category'], 'error_category', session_id))\n          for skill in skills[:5]:\n              if skill:\n                  db.execute('INSERT OR IGNORE INTO entities VALUES(?,?,?)',\n                             (skill, 'skill', session_id))\n          migrated += 1\n      except Exception as exc:\n          sys.stderr.write(f'recall migration: skipping {session_id}: {exc}\\n')\n          skipped += 1\n\n  db.execute('COMMIT')\n  # REBUILD OUTSIDE TRANSACTION: releases write lock before full table re-scan.\n  # Triggers already populated FTS5 during inserts; rebuild is idempotent but ensures\n  # optimal index structure. Moving outside COMMIT unblocks concurrent WAL readers.\n  db.execute(\"INSERT INTO sessions_fts(sessions_fts) VALUES('rebuild')\")\n  print(f'Migrated: {migrated}, Skipped (malformed): {skipped}')\n\nSTEP 5: MIGRATE failure_patterns AND learnings (separate transaction)\n  db.execute('BEGIN')\n  for pattern, failures_list in index.get('failure_patterns', {}).items():\n      for fl in failures_list:\n          db.execute('INSERT OR IGNORE INTO failure_patterns ...',\n                     (pattern, fl.get('cmd') or fl.get('command'), fl.get('error'),\n                      fl.get('resolution'), fl.get('session_id'), fl.get('date')))\n  for learning in index.get('learnings', []):\n      db.execute('INSERT OR IGNORE INTO learnings ...',\n                 (learning.get('bucket','personal'), learning.get('category','general'),\n                  learning.get('title',''), learning.get('fix'), learning.get('solution'),\n                  learning.get('session_id'), learning.get('date')))\n  db.execute('COMMIT')\n\nSTEP 6: WAL CHECKPOINT\n  try:\n      db.execute('PRAGMA wal_checkpoint(FULL)')\n  except sqlite3.OperationalError:\n      db.execute('PRAGMA wal_checkpoint(PASSIVE)')\n\nSTEP 7: WRITE v3 recall-index.json (atomic; full v2-format stubs; NO user_messages)\n  sorted_sessions = sorted(index.get('sessions',{}).items(),\n                            key=lambda x: x[1].get('date',''), reverse=True)\n  v3_sessions = {sid: s for sid, s in sorted_sessions[:50]}\n  v3_index = {\n      'version': 3, 'minimum_reader_version': '3.0.0', 'db': 'recall.db',\n      'project': index.get('project'),\n      'sessions': v3_sessions,\n      'failure_patterns': index.get('failure_patterns', {}),\n      'learnings': index.get('learnings', []),\n      'pending_learnings': index.get('pending_learnings', []),\n      'usage': index.get('usage', {}),\n  }\n  tmp = index_file.with_suffix('.json.tmp')\n  tmp.write_text(json.dumps(v3_index, indent=2, default=str))\n  os.replace(str(tmp), str(index_file))  # atomic\n\nSTEP 8: BACKUP\n  bak = index_file.with_name('recall-index.json.v2.bak')\n  if not bak.exists():  # never overwrite existing backup\n      bak.write_text(v2_backup_str)\n\nSTEP 9: VERIFY\n  count     = db.execute('SELECT COUNT(*) FROM sessions').fetchone()[0]\n  fts_count = db.execute('SELECT COUNT(*) FROM sessions_fts').fetchone()[0]\n  assert count == migrated, f'Session count mismatch: db={count} migrated={migrated}'\n  assert fts_count == count, f'FTS5 count mismatch: {fts_count} vs {count}'\n  # FTS5 tokenization smoke test (catches silent tokenizer failures COUNT(*) misses)\n  first_summary = next(iter(v3_sessions.values()), {}).get('summary', '')\n  if first_summary.strip():\n      test_term = next((w for w in first_summary.split() if len(w) >= 4), None)\n      if test_term:\n          # Just verify no exception; count may legitimately be 0 for rare tokens\n          db.execute(\n              'SELECT count(*) FROM sessions_fts '\n              'JOIN sessions s ON s.rowid=sessions_fts.rowid '\n              'WHERE sessions_fts MATCH ?', (test_term,)\n          ).fetchone()\n  # Rust binary forward-compatibility verification\n  import subprocess\n  rust_bin = Path(__file__).parent.parent / 'target' / 'release' / 'recall-sessions-rs'\n  if rust_bin.exists():\n      r = subprocess.run([str(rust_bin), str(get_project_dir(project_folder))],\n                         capture_output=True, text=True, timeout=10)\n      assert r.returncode == 0, f'Rust binary exited {r.returncode}: {r.stderr}'\n\nSTEP 10: REMOVE SENTINEL\n  sentinel.unlink()\n\nROLLBACK on any exception before sentinel removal:\n  try: db.rollback()\n  except: pass\n  db_path.unlink(missing_ok=True)\n  sentinel.unlink(missing_ok=True)\n  raise\n\nPost-migration documentation (add to ONBOARDING.md Maintenance section):\n  '/recall export after FTS5 migration produces a stub-only v3 JSON with 50 sessions.\n   For full backup: sqlite3 recall.db .dump > recall-full-backup.sql'\n\nMigration time: <3 seconds at 1000 sessions."

## Performance Targets

- **search_ms:** ~4ms at 1000 sessions (FTS5+UNION ~2ms + entity join ~0.5ms + Python scoring ~0.4ms + sidecar loads ~1ms). ~12ms at 10000 sessions. Well under the 200ms target.
- **session_start_ms:** ~17ms combined (Rust fast-path ~5ms + Python Phase 2 ~12ms). Under 200ms budget. Phase 2 is gated on Rust exit code 64 or RECALL_SESSION_START_VERBOSE=1.
- **session_end_s:** ~2s total SessionEnd budget (unchanged). SQLite write path adds ~50ms vs v2 JSON write. WAL checkpoint adds ~50ms with PASSIVE fallback. Migration (run once): <3s at 1000 sessions.
- **storage_per_session_kb:** ~3.5KB per session (sessions row ~550 bytes including failures_json blob, FTS5 external-content index overhead ~1.2KB, entities rows avg 7 at 40 bytes ~280 bytes, sidecar JSON ~1.5KB, recall-index.json stub ~250 bytes). At 1000 sessions: ~3.5MB. At 10000 sessions: ~35MB.
